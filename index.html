Pattern Guide

What these design patterns are solving

The goal is not to use patterns for style points. Each one solves a specific design pain: too much coupling, hard-coded behavior, noisy update flow, or difficult object construction.

Facade

Give the system one clean entry point

Facade hides internal complexity behind a smaller public API. In your Snake and Ladder example, gameManager acts as the coordinator so callers do not have to manually manage board setup, dice logic, rule checks, and player rotation.

Why we use it

It reduces the number of things the client has to know before it can use the system.

Problem it solves

Without it, the client becomes tightly coupled to too many low-level classes and the setup logic leaks everywhere.

class gameManager {
    Queue<User> playerQueue = new ArrayDeque<>();
    Board board;
    Play play;

    public gameManager(int boardId) {
        initBoard(boardId);
    }

    public void simulate(int playersCount) {
        // setup players
        // run the play loop
        // check winner
    }
}

Strategy

Swap algorithms without changing the client

Strategy isolates multiple behaviors behind one shared contract. In your Rate Limiter code, FixedWindowRateLimiter, SlidingWindow, and TokenBucket all implement the same RateLimiter interface.

Why we use it

It lets the client call the same method while changing the actual algorithm underneath.

Problem it solves

Without strategy, the client ends up with large if/else trees and hard-coded limiter logic that becomes difficult to extend.

interface RateLimiter {
    boolean isAllowRequest();
}

class FixedWindowRateLimiter implements RateLimiter {
    @Override
    public boolean isAllowRequest() { ... }
}

class TokenBucket implements RateLimiter {
    @Override
    public boolean isAllowRequest() { ... }
}

Observer

Notify interested objects when state changes

Observer is useful when one change should automatically inform many listeners. In Snake and Ladder, playerState notifies all registered players when somebody moves, instead of making each player keep checking manually.

Why we use it

It keeps the system event-driven and real-time, which is cleaner than repeated polling.

Problem it solves

Without observer, listeners must keep asking "did anything change yet?", which is inefficient and harder to test.

interface playerStatusObserver {
    void update(playerState data);
}

class playerState implements playerStatusSubject {
    private final List<playerStatusObserver> observers = new ArrayList<>();

    public void updatePosition(int pos, User user) {
        this.lastUpdated = user;
        this.lastPos = pos;
        notifyObserver();
    }
}

Builder

Make complex object creation easier to manage

You mentioned builders as a next step for custom boards or richer user setup. Builder is useful when object creation starts getting too many optional choices or too many constructor arguments.

Why we use it

It makes object creation readable when the configuration becomes more complex.

Problem it solves

Without builder, constructors become hard to read, hard to extend, and easy to misuse when there are many optional fields.

class BoardBuilder {
    private final Board board = new Board();

    public BoardBuilder withSnake(int from, int offset) { ... }
    public BoardBuilder withLadder(int from, int offset) { ... }
    public Board build() { return board; }
}
